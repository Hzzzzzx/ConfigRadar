Java / Spring 静态配置资产盘点

ConfigRadar 当前架构与数据流

ConfigRadar 的核心思路是：先把注解、配置文件、部署文件里的线索统一成稳定的配置事实清单，再基于这份清单做 HTML 报告、平台导出和版本 diff。

1. 当前代码现状

现在已经不是单一 detector，而是一条可扩展流水线。

输入层CLI 参数与扫描上下文

inventory、diff、config-diff、export 已成型，profile、include/exclude、rules、consumer 都通过上下文传入。

扫描层多 detector 静态扫描

覆盖 Spring 配置文件、Java 源码、metadata、XML、HOCON、Docker、Compose、Kubernetes，以及可选 codegraph 语义 detector。

模型层稳定配置事实模型

确定 key 进入 ConfigFinding，动态表达式进入 UncertainFinding，最终组合为 ConfigInventory。

消费层同一份 inventory 多种输出

YAML、HTML、default app_configs、XAC、diff 都从 inventory 派生，扫描和输出格式解耦。

2. 总体数据流

颜色含义：蓝色是入口，绿色是扫描，黄色是事实模型，紫色是输出，红色是 diff 审查点。

4. Inventory 是中心模型

所有输出都围绕这份结构，而不是围绕某个 detector 的临时结果。

结构含义典型字段
ConfigFinding确定 key 的配置事实key、normalizedKey、role、value、defaultValue、environment、source、confidence、detectorId
UncertainFinding有配置访问意图，但 key 无法静态确定expression、reason、rootSink、source、confidence
InventoryCheck自动审查提示敏感 key、远程配置中心入口、动态配置访问
Diagnostic扫描过程诊断某个 detector 失败时记录 warning，不拖垮整次扫描

5. Diff 场景数据流

diff 比较的是两份 inventory，不是源码文本。

base 源码
旧版本项目
→
扫描
base inventory
→
KeyBasedDiffStrategy
identity = normalizedKey + role + profile + region + namespace
←
扫描
head inventory
输出区块代表什么
added新版本新增的配置事实，比如新增了某个 key 的定义或读取。
removed旧版本存在但新版本消失的配置事实。
changed同一个 identity 下，value、defaultValue、source.path 等字段变化。
uncertainChanged新出现的动态配置访问，需要人工审查。
checksdiff 阶段补充的风险提示，比如新敏感 key 或新远程配置入口。

6. config-diff 的端到端流程

适合放进 CI 或提交流程：给两个 git ref，工具自己扫描并降噪。

base-ref / head-ref
用户输入两个提交点
→
临时 worktree
分别 checkout 两个版本并扫描
→
完整 diff
先按 inventory identity 得到完整配置变化
→
过滤输出
只保留 git changed files 相关配置项
java -jar config-radar-cli.jar config-diff \
  --repo ./my-service \
  --base-ref v1.2.0 \
  --head-ref main \
  -o changes.yaml

7. HTML 报告的位置

HTML 不是新的事实源，只是 inventory 的人类审查视图。

ConfigInventory
统一事实源
→
HtmlReportConsumer
按 key 聚合，选出 Spring 优先级下的生效值
→
config-report.html
统计卡、分布图、搜索、详情弹窗、uncertain、diagnostics
→
人工审查
上线配置准备、缺值补全、风险确认
java -jar config-radar-cli.jar inventory ./my-service \
  -o ./out/inventory.yaml \
  --consumer html
