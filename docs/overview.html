ConfigRadar 的配置数据流

这页只讲数据怎么流：原始配置长什么样，经过哪一步，最后变成什么配置产物。设计文档是主链路，代码扫描是兜底和校验。

1. 总体图：两条路径汇到同一个配置产物

主路径

→
转换规则字段映射、命名规范、敏感项分区
→
XAC 配置产物dist/xac/app-config.yaml
上线交付主产物
兜底路径
项目源码src/main/java
src/main/resources
→
扫描出的配置清单target/config-inventory.yaml
真实实现里的配置线索
→
校验结果漏项、默认值不一致、动态 key、敏感项
↘
修正 XAC 产物补齐或提醒人工确认
对外讲法：不是靠扫代码去猜最终配置。最终交付以设计文档转换出的 XAC 产物为主，代码扫描负责验证“设计里写的”和“代码里实际用的”是否一致。

2. 主路径：设计文档转成 XAC 产物

docs/config/payment-config.yaml
service: payment-service
configs:
  - key: payment.client.timeout
    value: "5000"
    scope: prod
    desc: 支付客户端超时时间
  - key: payment.client.token
    secret: true
    desc: 支付网关 token
dist/xac/app-config.yaml
data:
  app_configs:
    - config_key: payment.client.timeout
      config_value: "5000"
      secret: 0
  J2C:
    secrets:
      - key: payment_client_token
        password: "${payment_client_token}"
设计文档人写的配置清单
→
字段映射key/value/scope/secret
→
敏感项分区普通配置进 app_configs
密钥进 J2C
→
XAC YAML平台可接收产物

3000

src/main/resources/application-prod.yml

prod

来源：
application-prod.yml

→

→
和设计文档对账文档是否漏写？

5. 版本 Diff：对比的是两份配置清单

旧版本源码release-1.0
→
旧配置清单base inventory
↔
新配置清单head inventory
←
新版本源码main
Diff 输出新增配置、删除配置、值变化、动态项变化、敏感项提示
→
上线前审查这次发布到底改了哪些配置
重点：设计文档生成 XAC 是主链路；代码扫描生成清单，用来兜底校验和做版本差异审查。
