Java / Spring 配置资产盘点工具

ConfigRadar

不用运行应用,静态扫描出你的项目到底有哪些配置、定义在哪、读取在哪、有没有默认值、版本间改了什么。

1 它解决什么问题

一个真实痛点,每个 Java/Spring 团队都遇到过

❓

"这个项目上线到底需要配哪些东西?"

配置散落在各处:application.yml、.env、@Value、Environment.getProperty、Dockerfile、k8s yaml、远程配置中心……新人接手不知道该填哪些、运维不知道该准备哪些环境变量、上线后发现漏配了一个 key 直接炸。

ConfigRadar 把这些一次性盘清:扫源码和资源文件(不运行 app),产出一份结构化的配置清单,告诉你每个配置 key 的定义位置、读取位置、默认值、所属环境、以及哪些是动态的(需要人工确认)。

3 能扫到什么

8 个内置 detector,覆盖 Java/Spring 配置的主要来源

📄
Spring 配置文件
application*.yml、bootstrap*、.env、${...} 占位符、多文档 profile
☕
Java 源码
@Value(字段/参数/构造器)、@ConfigurationProperties、getProperty、System.getenv、常量拼接
🐳
部署文件
Dockerfile ENV、docker-compose、Kubernetes ConfigMap/Secret/env
🔧
配置中心
Apollo、Nacos、Spring Cloud Config 引用入口
📝
运行时 XML
web.xml、Logback/Log4j2 springProperty
📚
其他格式
HOCON、Spring Boot metadata JSON、PropertySource
⚠️ 诚实边界
扫的是静态资产(定义/读取位置 + 字面默认值),不是运行时实际值。打包 JAR 内部不扫(只扫源码/资源)。这些是后续阶段的范畴。

4 三个命令

一个 CLI 工具,三个核心动作

① inventory — 盘点配置

# 扫描项目,产出配置清单
java -jar config-radar-cli.jar inventory ./my-project -o inventory.yaml --profile prod

产出每个 key 的定义位置、读取位置、默认值、所属 profile,以及哪些是动态 key(归到 uncertain)。

② diff — 对比两个版本

# 扫两次,对比改动
java -jar config-radar-cli.jar inventory ./v1 -o before.yaml
java -jar config-radar-cli.jar inventory ./v2 -o after.yaml
java -jar config-radar-cli.jar diff --base before.yaml --head after.yaml -o diff.yaml

产出 added/removed/changed —— 上线前配置审查的核心动作。两个 inventory 用相同 profile,否则同一 key 会被误判成 add+remove。

③ export — 转成下游格式

# 转成配置中心格式(XAC 平台制品)
java -jar config-radar-cli.jar export --inventory inventory.yaml -o out.yaml --format xac

把 inventory 转成下游能直接用的格式。敏感 key 自动分区(占位符),缺值的单独列出供补全。

7 几个设计取舍

为什么这么做,而不是那样

扫描 vs 运行时

选择静态扫描:快、可重复、可 diff、不依赖运行环境。代价是拿不到运行时实际值(那是后续阶段)。对一个"配置资产盘点"工具,静态足够且更可靠。

上游不管下游

ConfigRadar 不内置模板引擎、不替下游决定输出格式。下游自己实现消费逻辑(怎么按模块/环境填字段、模板长什么样)。这让 ConfigRadar 永远不用背上"维护别人的下游逻辑"的包袱。

动态 key 不猜

遇到运行时拼接的 key(如 prefix + ".host"),能解析的解析(常量/字面量拼接),不能解析的诚实归到 uncertain,标明原因,而不是瞎猜一个值。

准确优先于覆盖率

宁可少报(明确漏了哪些),也不误报(把不是配置的当配置)。误报会侵蚀信任,漏报可以靠规则扩展补。
